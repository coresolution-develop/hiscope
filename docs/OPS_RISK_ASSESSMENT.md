# 운영 데이터 규모 가정 리스크 점검

## 데이터 규모 가정

- 기관 수: 50
- 기관당 직원 수: 2,000
- 기관당 연간 평가 배정: 100,000+
- 연간 업로드 이력: 30,000+
- 감사로그: 일 5만 건 수준 가능

## 리스크 평가

## 1) 권한/데이터 격리

- 리스크: URL 직접 접근 또는 ID 변조로 타 기관 데이터 노출
- 현재 대응:
  - 경로 권한(`SUPER_ADMIN`/`ORG_ADMIN`/`USER`) 분리
  - 서비스 레벨 `orgId` 검증
  - org 기준 repository 조회 사용
- 잔여 리스크:
  - 신규 기능 개발 시 `organization_id` 조건 누락 가능
- 추가 권고:
  - 컨트롤러/서비스 보안 회귀 테스트를 릴리즈 게이트에 포함

## 2) 업로드 실패 처리

- 리스크: 파싱 실패/예외 발생 시 이력 미기록으로 운영 추적 불가
- 현재 대응:
  - 업로드 성공/부분/실패 이력 저장
  - 비즈니스 예외 및 시스템 예외 모두 실패 이력 기록
- 개선 반영:
  - 문항 업로드 포함 전 업로드 경로에서 실패 이력 저장 보장

## 3) 대량 업로드/메모리 사용

- 리스크: 대용량 xlsx로 인한 메모리 압박/응답 지연
- 현재 대응:
  - 파일 크기 제한, 행수 제한, 확장자 제한
  - 실패 즉시 반환
- 잔여 리스크:
  - 단일 트랜잭션 대량 저장의 지연
- 추가 권고:
  - 장기적으로 비동기 업로드 큐 + 배치 flush 전략 도입

## 4) 대량 조회 성능

- 리스크: 감사로그/이력/목록 조회 시 응답 저하
- 현재 대응:
  - 페이지네이션, 주요 인덱스 추가
  - 화면 조회 건수 상한 설정
- 잔여 리스크:
  - 복합 검색 조건 증가 시 인덱스 재조정 필요
- 추가 권고:
  - 슬로우 쿼리 로그 기반 인덱스 튜닝 주기 운영

## 5) 감사 추적 신뢰성

- 리스크: 프록시 환경에서 실제 클라이언트 IP 유실
- 현재 대응:
  - Request ID, DB 감사로그 + 파일 감사로그 이중 저장
- 개선 반영:
  - `X-Forwarded-For`/`X-Real-IP` 우선 사용, 길이 제한 적용
  - 감사로그 트랜잭션 분리로 롤백 영향 최소화

## 6) 운영 설정 분리 실패

- 리스크: 로컬 시드/디버그 설정이 운영에 유입
- 현재 대응:
  - profile 분리(local/prod/test)
  - migration 경로 분리(common/local)
  - bootstrap super admin 기본 비활성화
- 추가 권고:
  - 배포 파이프라인에서 `prod` 설정 정적 검증(필수 env 누락 차단)

## 7) 배포/롤백 절차 미흡

- 리스크: 배포 성공 여부 판단 지연, 장애 시 롤백 실패
- 현재 대응:
  - 배포 전 체크리스트 문서화
  - health check/로그 확인 절차 존재
- 추가 권고:
  - 배포 자동화 스크립트에 health check timeout + 자동 롤백 조건 포함

## 우선순위 결론

- 즉시 조치 완료: 업로드 실패 이력 보장, 업로드 행수 제한, 프록시 IP 감사로그 반영
- 운영 직전 필수: UAT 증적 확보, 배포 체크리스트 전항목 완료
- 중기 개선: 비동기 업로드, 성능 관측 고도화, 보안 회귀 테스트 자동화
